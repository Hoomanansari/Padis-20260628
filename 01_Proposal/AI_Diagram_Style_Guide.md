# AI Diagram & Image Design Standard

## هدف

این سند، استاندارد رسمی طراحی، تولید، نگهداری و استفاده از تمامی تصاویر، دیاگرام‌ها، فلوچارت‌ها، معماری‌ها، اینفوگرافیک‌ها و تصاویر مفهومی پروژه Padis / Syncolony را تعریف می‌کند.

هدف این استاندارد، ایجاد هویت بصری یکپارچه در کل پروپوزال است تا تمامی تصاویر، صرف‌نظر از زمان تولید یا ابزار تولید، دارای ظاهر، کیفیت، ساختار، نام‌گذاری و قابلیت ویرایش یکسان باشند.

این سند جایگزین نسخه قبلی `AI_Diagram_Style_Guide.md` می‌شود و باید به عنوان مرجع اصلی تولید همه تصاویر پروژه استفاده شود.

## 1. اصول کلی

تمام تصاویر باید:

- حرفه‌ای و مناسب ارائه به مدیران ارشد باشند.
- از نظر بصری ساده، خوانا و بدون شلوغی باشند.
- قابلیت چاپ با کیفیت بالا در Word و PDF را داشته باشند.
- برای استفاده در نسخه فارسی و نسخه انگلیسی مناسب باشند.
- مستقل از برند تولیدکننده تصویر باشند.
- در تمام پروپوزال دارای هویت بصری یکسان باشند.
- قابل ویرایش، قابل بازتولید و قابل نگهداری در Repository باشند.
- ابتدا به صورت برداری و استاندارد تولید شوند و سپس در صورت نیاز خروجی PNG از همان فایل برداری ایجاد شود.

## 3. رنگ‌های استاندارد

رنگ‌های اصلی پروژه به شرح زیر است:

| کاربرد | رنگ | کد رنگ |
|---|---|---|
| Primary Blue | آبی اصلی | `#005BAC` |
| Secondary Blue | آبی ثانویه | `#4A90E2` |
| Dark Gray | خاکستری تیره | `#4A4A4A` |
| Light Gray | خاکستری روشن | `#D9D9D9` |
| White | سفید | `#FFFFFF` |
| Success | سبز موفقیت | `#2E7D32` |
| Warning | نارنجی هشدار | `#F5A623` |
| Alarm | قرمز آلارم | `#D32F2F` |

قواعد استفاده از رنگ:

- حداکثر از ۶ رنگ اصلی در هر تصویر استفاده شود.
- رنگ‌ها باید یکنواخت و تکرارپذیر باشند.
- از Gradient شدید استفاده نشود.
- رنگ قرمز فقط برای Alarm، Error یا مسیرهای هشدار استفاده شود.
- رنگ نارنجی فقط برای Warning یا وضعیت نیازمند توجه استفاده شود.
- رنگ سبز فقط برای وضعیت مجاز، موفق یا تأییدشده استفاده شود.

## 15. استاندارد فایل SVG

تمام دیاگرام‌های پروژه باید ابتدا به صورت Native SVG تولید شوند.

قواعد الزامی SVG:

- فایل SVG باید استاندارد و معتبر باشد.
- فایل SVG باید در Inkscape بدون خطا باز شود.
- متن‌های داخل SVG باید در Inkscape قابل ویرایش باشند.
- اجزای اصلی تصویر باید از عناصر برداری SVG تشکیل شده باشند.
- فایل SVG نباید صرفاً تبدیل‌شده از PNG یا JPG باشد.
- فایل SVG نباید شامل تصویر raster به صورت embedded یا base64 به عنوان محتوای اصلی دیاگرام باشد.
- استفاده محدود از تصویر raster فقط برای موارد خاص و با دلیل مشخص مجاز است؛ اما برای دیاگرام‌های معماری، فلوچارت‌ها و اینفوگرافیک‌ها ممنوع است.
- Textها نباید به Path تبدیل شوند، مگر برای نسخه نهایی غیرقابل ویرایش.
- فایل SVG باید قابل ویرایش در Inkscape، Figma، Adobe Illustrator یا ابزارهای مشابه باشد.
- فایل SVG باید قابل نگهداری و بررسی در GitHub باشد.

## 16. استاندارد تولید PNG از SVG

برای هر تصویر، فایل PNG باید از روی همان فایل SVG مرجع تولید شود.

قواعد خروجی PNG:

- PNG نباید جداگانه و مستقل از SVG توسط AI تولید شود.
- PNG و SVG باید از نظر ظاهر یکسان باشند.
- اگر SVG و PNG تفاوت ظاهری قابل توجه داشته باشند، PNG معتبر نیست و باید دوباره از SVG خروجی گرفته شود.
- PNG برای استفاده در Word و PDF استفاده می‌شود.
- SVG فایل مرجع و قابل ویرایش پروژه است.

کیفیت PNG:

- حداقل 300 DPI برای خروجی‌های چاپی
- رزولوشن مناسب برای درج در Word
- حفظ خوانایی متن‌ها در اندازه واقعی سند

## 18. استاندارد نام‌گذاری فایل‌ها

فرمت استاندارد نام‌گذاری:

```text
<Chapter>_<Section>_<Figure>_<Short_Name>
```

تعریف اجزا:

| بخش | معنی |
|---|---|
| Chapter | شماره فصل اصلی سند |
| Section | شماره زیر‌بخش |
| Figure | شماره تصویر در همان زیر‌بخش |
| Short_Name | عبارت کوتاه و توصیفی برای تصویر |

مثال:

```text
04_05_01_Integrated_Solution_Architecture
```

توضیح مثال:

- `04` یعنی تصویر مربوط به فصل یا بخش اصلی 04 است.
- `05` یعنی تصویر مربوط به زیر‌بخش 05 است.
- `01` یعنی اولین تصویر استفاده‌شده در آن زیر‌بخش است.
- `Integrated_Solution_Architecture` عبارت کوتاه توصیفی تصویر است.

نمونه‌های مجاز:

```text
04_05_01_Integrated_Solution_Architecture
04_06_01_Software_Solution_Architecture
05_02_01_Student_Access_Flow
05_03_01_Visitor_Management
07_03_02_Parking_Control_Process
11_01_01_Parking_Architecture
```

قواعد نام‌گذاری:

- از فاصله Space استفاده نشود.
- از Underscore `_` برای جداسازی استفاده شود.
- از حروف انگلیسی استفاده شود.
- عبارت توصیفی باید کوتاه، روشن و قابل فهم باشد.
- شماره فایل باید با شماره شکل و محل استفاده در سند هماهنگ باشد.
- اگر تصویر دوم در همان زیر‌بخش اضافه شود، شماره Figure به `02` تغییر کند.
- اگر تصویر به زیر‌بخش دیگری مربوط است، شماره Section باید مطابق همان زیر‌بخش باشد.

## 19. شماره‌گذاری شکل‌ها در سند

شماره تصویر در Word یا Proposal باید با نام فایل هماهنگ باشد.

مثال:

نام فایل:

```text
04_06_01_Software_Solution_Architecture.png
```

عنوان شکل در سند:

```text
شکل 4-6-1 – معماری مفهومی راهکار نرم‌افزاری یکپارچه
```

قواعد:

- شماره شکل باید محل استفاده تصویر در سند را نشان دهد.
- تغییر محل تصویر در سند باید با به‌روزرسانی نام فایل و Caption همراه باشد.
- هر تصویر باید Caption رسمی داشته باشد.
- Caption باید کوتاه، توصیفی و قابل استفاده در فهرست اشکال باشد.

## 24. قاعده تولید توسط AI

در هنگام استفاده از هوش مصنوعی برای طراحی یا کمک به تولید تصویر، موارد زیر باید رعایت شود:

- AI نباید خروجی نهایی SVG غیرقابل ویرایش تولید کند.
- تصویر نهایی باید به صورت Native SVG استاندارد قابل ویرایش باشد.
- از همین سند به عنوان مرجع طراحی استفاده شود.
- سبک تمام تصاویر باید با تصاویر قبلی کاملاً یکسان باشد.
- در صورت تضاد بین زیبایی و خوانایی، خوانایی اولویت دارد.
- از تولید تصاویر شلوغ یا دارای جزئیات غیرضروری خودداری شود.
- از تولید تصویر با متن‌های طولانی داخل خود تصویر خودداری شود.
- اگر خروجی AI به صورت PNG تولید شد، آن PNG فقط می‌تواند برای ایده‌پردازی یا مرجع بصری استفاده شود، نه به عنوان فایل مرجع نهایی.
- تصویر نهایی باید قابلیت ویرایش و بازتولید در آینده را داشته باشد.

- SVG شامل PNG/JPG embedded به عنوان محتوای اصلی نباشد.
- PNG از همان SVG خروجی گرفته شده باشد.
- ظاهر PNG و SVG یکسان باشد.
- متن‌ها در اندازه واقعی Word خوانا باشند.
- Canvas مطابق استاندارد تعیین‌شده باشد.
- نام فایل مطابق استاندارد نام‌گذاری باشد.
- Caption پیشنهادی تصویر مشخص باشد.
- مسیر ذخیره فایل مطابق ساختار Repository باشد.

اگر هر یک از موارد بالا برقرار نباشد، تصویر نباید به عنوان نسخه نهایی Commit شود.

## 26. قاعده نگهداری در Repository

تمام تصاویر در مسیر زیر نگهداری شوند:

```text
01_Proposal/Images/
```

برای هر تصویر، فایل‌های زیر نگهداری شوند:

- فایل SVG به عنوان نسخه مرجع و قابل ویرایش
- فایل PNG به عنوان نسخه قابل استفاده در Word و PDF
- در صورت نیاز، فایل PDF برداری برای چاپ

پیشنهاد تکمیلی برای آینده:

```text
01_Proposal/Images/SVG/
01_Proposal/Images/PNG/
01_Proposal/Images/Icons/
01_Proposal/Templates/
```

تا زمانی که ساختار نهایی Repository تغییر نکرده است، مسیر فعلی `01_Proposal/Images/` معتبر است.

## 27. Template و Icon Library

Templateها و Icon Library بخشی از خود Style Guide نیستند، اما باید با این سند سازگار باشند.

پیشنهاد می‌شود در ادامه پروژه فایل‌های زیر ایجاد شوند:

```text
01_Proposal/Templates/Diagram_Template.svg
01_Proposal/Images/Icons/
```

کاربردها:

- `Diagram_Template.svg` قالب پایه برای همه دیاگرام‌های جدید خواهد بود.
- `Icons/` شامل آیکون‌های استاندارد پروژه خواهد بود.
- همه دیاگرام‌ها باید از همین Template و Icon Library استفاده کنند تا ظاهر پروژه یکپارچه بماند.

این موارد باید به عنوان Asset پروژه نگهداری شوند، نه به عنوان متن داخل Style Guide.

# AI Diagram & Image Design Standard

## هدف

این سند، استاندارد رسمی طراحی تمامی تصاویر، دیاگرام‌ها، فلوچارت‌ها، معماری‌ها، اینفوگرافیک‌ها و تصاویر مفهومی پروژه Padis / Syncolony را تعریف می‌کند.

هدف این استاندارد، ایجاد هویت بصری یکپارچه در کل پروپوزال است تا تمامی تصاویر، صرف‌نظر از زمان تولید یا ابزار تولید (AI یا طراحی دستی)، دارای ظاهر، کیفیت، ساختار و قابلیت ویرایش یکسان باشند.

---

## اصول کلی

تمام تصاویر باید:

- حرفه‌ای و مناسب ارائه به مدیران ارشد باشند.
- از نظر بصری ساده، خوانا و بدون شلوغی باشند.
- قابلیت چاپ با کیفیت بالا در Word و PDF را داشته باشند.
- برای استفاده در نسخه فارسی و نسخه انگلیسی مناسب باشند.
- مستقل از برند تولیدکننده تصویر باشند.
- در تمام پروپوزال دارای هویت بصری یکسان باشند.
- دارای فایل مرجع قابل ویرایش باشند.
- در آینده توسط تیم پروژه، بدون نیاز به تولید مجدد کامل، قابل اصلاح باشند.

## رنگ‌های استاندارد

رنگ‌های اصلی پروژه:

| کاربرد | رنگ | کد |
|---|---|---|
| Primary Blue | آبی اصلی | `#005BAC` |
| Secondary Blue | آبی مکمل | `#4A90E2` |
| Dark Gray | خاکستری تیره | `#4A4A4A` |
| Light Gray | خاکستری روشن | `#D9D9D9` |
| White | سفید | `#FFFFFF` |
| Success | سبز وضعیت مجاز | `#2E7D32` |

رنگ‌ها باید کاربرد معنایی ثابت داشته باشند. برای مثال، قرمز فقط برای هشدار یا رخداد امنیتی استفاده شود و سبز برای وضعیت مجاز یا تجهیزات عملیاتی.

## استاندارد متن داخل تصاویر

متن داخل تصاویر باید فقط برای برچسب‌گذاری استفاده شود، نه توضیح دادن.

قواعد اصلی:

- حداکثر دو یا سه کلمه برای هر عنصر.
- از جمله‌های توضیحی استفاده نشود.
- از پاراگراف استفاده نشود.
- از متن‌های طولانی داخل تصویر استفاده نشود.
- توضیحات مفهومی، فنی و تشریحی باید در متن سند Word / Markdown نوشته شود، نه داخل تصویر.
- تمام متن‌ها باید خوانا باشند.
- اندازه فونت باید در خروجی Word و PDF قابل خواندن باشد.
- متن‌ها باید در فایل SVG به صورت Text واقعی باقی بمانند و تا حد امکان به Path تبدیل نشوند.

نمونه مناسب داخل تصویر:

- کنترل دسترسی
- مدیریت پارکینگ
- پایگاه داده
- سامانه‌های بیرونی
- هشدار و رخدادها

نمونه نامناسب داخل تصویر:

- این ماژول برای مدیریت کامل ورود و خروج کاربران استفاده می‌شود.
- سامانه مرکزی اطلاعات را از تجهیزات دریافت کرده و سپس تحلیل می‌کند.
- در صورت ورود غیرمجاز، هشدار برای اپراتور ارسال می‌شود.

## استاندارد تولید و نگهداری فایل‌های SVG

تمام دیاگرام‌های پروژه باید ابتدا به‌صورت Native SVG طراحی شوند.

فایل SVG باید:

- یک SVG استاندارد و معتبر باشد.
- در Inkscape بدون خطا باز شود.
- اجزای اصلی آن شامل خطوط، اشکال، آیکون‌ها و متن‌ها به صورت برداری باشند.
- متن‌ها در آن قابل ویرایش باشند.
- برای ویرایش در Inkscape، Figma، Adobe Illustrator یا ابزارهای مشابه مناسب باشد.
- در GitHub به عنوان فایل مرجع قابل نگهداری باشد.

فایل SVG نباید:

- صرفاً تبدیل‌شده از PNG یا JPG باشد.
- شامل یک تصویر raster embedded/base64 به عنوان محتوای اصلی دیاگرام باشد.
- فقط یک Screenshot داخل SVG باشد.
- متن‌ها را بدون ضرورت به Path تبدیل کرده باشد.
- در Inkscape با علامت خطا یا ضربدر قرمز نمایش داده شود.
- وابسته به افکت‌هایی باشد که در Word یا PDF درست رندر نمی‌شوند.

### متن در SVG

برای حفظ قابلیت ویرایش:

- متن‌ها باید به صورت `<text>` یا Text Object واقعی باقی بمانند.
- تبدیل متن به Path فقط برای نسخه نهایی غیرقابل ویرایش مجاز است، نه برای فایل مرجع پروژه.
- در صورت استفاده از فونت خاص، نام فونت باید در سند یا توضیح فایل مشخص باشد.
- برای نسخه قابل ویرایش، اولویت با خوانایی و قابلیت اصلاح است.

### تولید PNG از SVG

PNG باید از همان فایل SVG مرجع خروجی گرفته شود.

ممنوع است:

- SVG و PNG جداگانه با AI تولید شوند.
- PNG از یک تصویر تولید شود و SVG بعداً به‌صورت تقریبی Trace شود.
- ظاهر PNG و SVG با هم متفاوت باشد.

قاعده اصلی:

> SVG منبع اصلی است و PNG فقط خروجی رندرشده از همان SVG است.

## نام‌گذاری فایل‌ها

فرمت استاندارد نام‌گذاری:

```text
<Chapter>_<Section>_<Figure>_<Short_Description>
```

تعریف اجزا:

- `Chapter`: شماره فصل اصلی سند.
- `Section`: شماره زیر‌بخش.
- `Figure`: شماره تصویر در همان زیر‌بخش.
- `Short_Description`: عبارت کوتاه توصیفی برای تصویر.

مثال:

```text
04_05_01_Integrated_Solution_Architecture
```

در این مثال:

- `04` یعنی تصویر مربوط به بخش یا فصل 04 است.
- `05` یعنی تصویر مربوط به زیر‌بخش 05 است.
- `01` یعنی اولین تصویر استفاده‌شده در آن زیر‌بخش است.
- `Integrated_Solution_Architecture` نام کوتاه و توصیفی تصویر است.

نمونه‌های معتبر:

```text
04_05_01_Integrated_Solution_Architecture.svg
04_05_01_Integrated_Solution_Architecture.png
04_06_01_Software_Solution_Architecture.svg
04_06_01_Software_Solution_Architecture.png
05_02_01_Student_Access_Flow.svg
05_02_01_Student_Access_Flow.png
11_01_01_Parking_Architecture.svg
11_01_01_Parking_Architecture.png
```

قواعد نام‌گذاری:

- از Space استفاده نشود.
- از حروف انگلیسی استفاده شود.
- از Underscore برای جداسازی کلمات استفاده شود.
- نام فایل باید کوتاه، دقیق و قابل فهم باشد.
- نام فایل نباید شامل کاراکترهای فارسی، علامت‌های خاص یا تاریخ تولید باشد.

## قاعده استفاده در Word و PDF

برای Word:

- استفاده از SVG در نسخه‌های جدید Microsoft Word امکان‌پذیر است.
- با این حال، برای کنترل ظاهری و سازگاری بیشتر، نسخه PNG نیز باید نگهداری شود.
- در فایل Word نهایی، بسته به سازگاری نسخه Office، می‌توان از SVG یا PNG استفاده کرد.

برای PDF:

- خروجی PDF باید بررسی شود.
- اگر SVG در PDF به‌درستی رندر نشد، از PNG با کیفیت بالا استفاده شود.
- در هر حالت، فایل SVG مرجع باید در Repository باقی بماند.

- [ ] فایل SVG وجود دارد.
- [ ] فایل PNG وجود دارد.
- [ ] SVG در Inkscape بدون خطا باز می‌شود.
- [ ] متن‌های SVG قابل ویرایش هستند.
- [ ] PNG از همان SVG تولید شده است.
- [ ] ظاهر SVG و PNG یکسان است.
- [ ] تصویر فاقد متن توضیحی طولانی است.
- [ ] تصویر از رنگ‌های استاندارد پروژه استفاده می‌کند.
- [ ] تصویر دارای پس‌زمینه سفید است.
- [ ] تصویر در Word خوانا است.
- [ ] تصویر در PDF خوانا است.

# معماری کلان سیستم پیشنهادی

## هدف سند

این سند معماری کلان سیستم پیشنهادی برای پروژه مدرسه را تشریح می‌کند. هدف از این معماری، ایجاد یک چارچوب فنی و قابل ارائه برای کنترل تردد، کنترل دسترسی، مدیریت امنیت، مدیریت مهمانان و مدیریت پارکینگ بر اساس راهکارهای پیشنهادی HLWL و محصولات Syncolony است.

این سند در مرحله فعلی، معماری مفهومی و پیشنهادی را ارائه می‌کند و وارد جزئیات نهایی مدل تجهیزات، تعداد قطعی تجهیزات، قیمت‌گذاری یا طراحی اجرایی کامل نمی‌شود. جزئیات نهایی پس از تحلیل پلان، بررسی زیرساخت، بازدید میدانی و تأیید کارفرما مشخص خواهد شد.

---

## 1. اصل طراحی معماری

اصل کلیدی معماری پیشنهادی این پروژه به شرح زیر است:

> هر فرد، وسیله نقلیه یا مراجعه‌کننده باید فقط در زمان مجاز، مکان مجاز و با سطح دسترسی مجاز امکان تردد داشته باشد.

بنابراین معماری سیستم فقط شامل تجهیزات فیزیکی مانند گیت و قفل نیست، بلکه یک معماری چندلایه شامل نرم‌افزار، سیاست‌های دسترسی، تجهیزات میدانی، مانیتورینگ، گزارش‌گیری، هشدار و یکپارچه‌سازی با سامانه‌های مدرسه است.

---

## 2. اهداف معماری پیشنهادی

معماری پیشنهادی باید اهداف زیر را پوشش دهد:

- کنترل متمرکز ورود و خروج افراد
- کنترل دسترسی به کلاس‌ها، آزمایشگاه‌ها، اتاق‌های اداری و فضاهای حساس
- کنترل تردد دانش‌آموزان براساس زمان، کلاس، بخش و برنامه آموزشی
- مدیریت دسترسی پرسنل، معلمان، مدیران و نیروهای خدماتی
- مدیریت ورود والدین، مهمانان و مراجعان
- کنترل خودروها و پارکینگ
- ثبت و گزارش‌گیری از رخدادهای تردد
- ایجاد هشدار در رخدادهای غیرمجاز یا مشکوک
- امکان اتصال به سامانه‌های مدرسه از طریق API یا Web Service
- امکان توسعه مرحله‌ای سیستم در فازهای بعدی
- استفاده از پلان‌های پروژه برای جانمایی مفهومی تجهیزات

### گروه‌های خودرویی

## 5. لایه احراز هویت و شناسایی

این لایه وظیفه شناسایی فرد یا خودرو را قبل از صدور مجوز تردد بر عهده دارد.

### روش‌های پیشنهادی برای افراد

- تشخیص چهره
- کارت RFID
- PIN یا کد دسترسی، در صورت نیاز
- QR یا کارت موقت برای مهمانان
- اثر انگشت به عنوان Option، در صورت درخواست کارفرما

### روش‌های پیشنهادی برای خودروها

- ANPR / پلاک‌خوان
- کارت RFID
- تگ UHF
- ثبت دستی یا تأیید اپراتور برای موارد خاص

### رویکرد پیشنهادی

برای ساده‌سازی تصمیم‌گیری مشتری، در نسخه اولیه پروپوزال بهتر است یک راهکار اصلی معرفی شود و روش‌های جایگزین به عنوان Option ارائه شوند. برای مثال:

- راهکار اصلی نفرات: تشخیص چهره + RFID
- راهکار اصلی خودروها: ANPR + UHF
- راهکار مهمان: ثبت در Reception + کارت موقت یا QR

این لایه، مغز تصمیم‌گیری سیستم است و تعیین می‌کند چه کسی، در چه زمانی، از چه نقطه‌ای و به کدام فضا اجازه ورود دارد.

### معیارهای تصمیم‌گیری

- نقش کاربر
- گروه کاربری
- زمان و روز هفته
- مکان یا ناحیه
- نوع فضا
- برنامه آموزشی
- وضعیت مجوز
- وضعیت مهمان یا پیمانکار
- وضعیت خودرو
- شرایط اضطراری

### نمونه سیاست‌ها

| سناریو | سیاست نمونه |
|---|---|
| دانش‌آموز | فقط در ساعات مجاز وارد مدرسه شود |
| دانش‌آموز | فقط به کلاس تعریف‌شده خود دسترسی داشته باشد |
| دانش‌آموز | فقط در زمان برنامه‌شده وارد آزمایشگاه شود |
| معلم | به کلاس‌ها و آزمایشگاه‌های مرتبط دسترسی داشته باشد |
| پرسنل اداری | فقط به بخش‌های اداری مجاز دسترسی داشته باشد |
| مهمان | فقط در بازه زمانی و مسیر تعریف‌شده تردد کند |
| خودرو پرسنل | فقط از ورودی پارکینگ مجاز وارد شود |
| پیمانکار | فقط در زمان قرارداد و محل مشخص دسترسی داشته باشد |

- مدیریت هشدارها

- سامانه برنامه هفتگی کلاس‌ها
- سامانه والدین
- Active Directory
- CCTV / VMS
- Fire Alarm System
- سیستم اطلاع‌رسانی
- سیستم گزارش‌گیری مدیریتی

### جریان‌های یکپارچه‌سازی پیشنهادی

| جریان | شرح |
|---|---|
| دریافت اطلاعات دانش‌آموزان | دریافت اطلاعات پایه، کلاس، پایه و وضعیت فعال/غیرفعال |

| ارسال رخدادهای تردد | ارسال ورود/خروج به سامانه مدرسه یا پنل والدین |
| اتصال به اعلام حریق | آزادسازی درب‌ها و گیت‌ها در وضعیت اضطراری |

## 16. وابستگی‌های زیرساختی معماری

### شبکه

برای هر تجهیز IP باید وضعیت شبکه بررسی شود:

- Face Terminal
- IP Camera
- Access Controller
- Gate Controller
- ANPR Camera
- Parking Controller
- Workstation
- Server

موارد موردنیاز:

- نود شبکه
- کابل‌کشی
- PoE یا برق مستقل
- رک شبکه
- VLAN امنیتی
- پهنای باند کافی
- دسترسی پشتیبانی

### برق

برای تجهیزات زیر برق پایدار لازم است:

- گیت‌ها
- قفل‌ها
- کنترلرها
- ترمینال‌های تشخیص چهره
- Barrier Gate
- دوربین‌ها
- سرور و تجهیزات شبکه

### سرور

نیاز سرور باید بر اساس موارد زیر تعیین شود:

- تعداد کاربران

- نیاز به ذخیره تصویر
- نیاز به گزارش‌گیری
- نوع نصب On-Premise، Cloud یا Hybrid
